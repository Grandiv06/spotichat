# SpotiChat — Backend Design & Implementation Prompt

این سند یک پرامپت کامل برای طراحی و پیاده‌سازی بک‌اند SpotiChat است. لطفاً با Claude Opus 4.6 (thinking) آن را بخوان و بک‌اند را طبق مشخصات زیر طراحی و پیاده کن. هر جا فرانت نیمه‌کاره است، هم فرانت و هم بک‌اند را تکمیل کن.

---

## 1. خلاصهٔ پروژه

**SpotiChat** یک اپلیکیشن چت/مسنجر وب با UX شبیه تلگرام است:
- احراز هویت با شماره تلفن + OTP
- لیست چت‌ها + ردیف استوری + ویو چت با ریپلای، ری‌اکشن، پیام صوتی/تصویری
- تنظیمات (پروفایل، مخاطبین، اعلان‌ها، حریم خصوصی و امنیت، دستگاه‌ها، ذخیره‌سازی، تم، کمک، خروج)

**وضعیت فعلی:** فقط فرانت‌اند؛ همهٔ داده‌ها از سرویس‌های mock (بدون HTTP/WebSocket) می‌آیند.

---

## 2. استک فرانت‌اند (فعلی)

- **Framework:** React 19 + Vite 7 + TypeScript 5.9 + React Router 7
- **State:** Zustand (auth, chat, modal, theme, settings, privacy)
- **فرم‌ها:** react-hook-form + zod + @hookform/resolvers
- **UI:** Tailwind CSS 4، کامپوننت‌های شادسی‌ان (shadcn)، lucide-react، فونت Geist
- **مسیرها:** `/login`, `/otp`, `/` (چت اصلی)

---

## 3. قابلیت‌های قابل دسترسی (وضعیت هر کدام)

### 3.1 احراز هویت (Login / OTP / Session)

**قابل دسترسی:** بله — UI کامل.

- **صفحه لاگین:** انتخاب کد کشور + شماره تلفن (پشتیبانی از اعداد فارسی)، نرمال‌سازی به digits، ارسال به سرویس mock.
- **صفحه OTP:** ۵ باکس جدا برای کد، paste، اعتبارسنجی zod (۵ رقم)، تایمر ارسال مجدد، بعد از تأیید درخواست دسترسی میکروفون/دوربین و ذخیرهٔ ترجیح در localStorage.
- **سرویس فعلی (mock):** `auth.service.ts` — `sendOtp(phone)` فقط لاگ می‌زند و `verifyOtp(phone, code)` فقط وقتی `code === '12345'` کاربر mock برمی‌گرداند؛ هیچ HTTP/WebSocket نیست.
- **Session:** فقط Zustand با persist به localStorage (`auth-storage`); توکن/رفرش واقعی وجود ندارد.

**درخواست برای بک‌اند:**
- لاگین با استانداردترین روش (مثلاً JWT + refresh token یا session معتبر).
- فعلاً **۳ شماره دستی** در سیستم تعریف شود و برای **همهٔ این شماره‌ها کد OTP ثابت `12345`** باشد تا تست آسان شود.
- بعد از verify، session/token برگردانده شود و فرانت آن را ذخیره و در درخواست‌ها بفرستد.

---

### 3.2 پروفایل (Profile)

**قابل دسترسی:** بله — ویو و ویرایش.

- **تنظیمات > My Profile:** نمایش آواتار، نام، آنلاین، شماره، یوزرنیم، بیو.
- **حالت ویرایش:** فیلدهای First Name، Username، Bio قابل ویرایش و ذخیره در `auth.store` (و در نتیجه localStorage)؛ **آپلود آواتار پیاده نشده** — دکمه/اوورلی «Set New Photo» فقط ظاهری است و هیچ انتخاب فایل یا API آپلود ندارد.

**درخواست برای بک‌اند و فرانت:**
- API برای به‌روزرسانی پروفایل (نام، یوزرنیم، بیو).
- API آپلود عکس پروفایل (مثلاً multipart به بک‌اند یا لینک از سرویس فایل)؛ در فرانت باید انتخاب فایل و ارسال به API اضافه شود (تکمیل نیمه‌کاره).

---

### 3.3 مخاطبین (Contacts)

**قابل دسترسی:** بله — دو جا.

- **تنظیمات > Contacts:** لیست از `mockedContacts` (فایل mock)، جستجو، مشاهده کارت مخاطب، افزودن/ویرایش در دیالوگ؛ **همه فقط state محلی**؛ حتی به contactService وصل نیست.
- **مودال‌های Add Contact و Search User (از sidebar/چت):** از `contact.service.ts` استفاده می‌کنند که روی آرایهٔ درون‌حافظهٔ `mockUsersDatabase` جستجو/افزودن می‌کند؛ هیچ API واقعی نیست.

**درخواست برای بک‌اند و فرانت:**
- API لیست مخاطبین کاربر، جستجو کاربر (برای اضافه کردن به مخاطبین)، اضافه/حذف مخاطب.
- در فرانت، صفحهٔ Contacts در تنظیمات باید به این APIها وصل شود تا داده واقعی نشان دهد و افزودن/ویرایش روی سرور ذخیره شود.

---

### 3.4 اعلان‌ها (Notifications)

**قابل دسترسی:** بله — فقط UI و رفتار محلی.

- **تنظیمات > Notifications:** Enable Notifications، Show Message Preview، Mute All Chats، Mute Group Chats؛ Message Sound (لیست صداها با پیش‌نمایش از `lib/sounds.ts`)؛ Incoming Call Sound.
- دادهٔ اولیه از `mockedNotificationSettings`؛ بعد از تغییر فقط در state کامپوننت و برای صدا در localStorage ذخیره می‌شود؛ هیچ sync با سرور نیست.

**درخواست برای بک‌اند:**
- ترجیحات اعلان را در پروفایل/تنظیمات کاربر در دیتابیس ذخیره کن و API برای خواندن/نوشتن این تنظیمات ارائه کن. فرانت می‌تواند بعداً به این API وصل شود.

---

### 3.5 حریم خصوصی و امنیت (Privacy & Security)

**قابل دسترسی:** بله — بخشی کامل، بخشی نیمه‌کاره.

- **Privacy (گزینه‌ها):**  
  - Phone Number / Last Seen & Online / Profile Photo هر کدام با سه گزینه: Everybody، My Contacts، Nobody و در صورت نیاز «استثناها» یا «همیشه اجازه» با انتخاب مخاطبین از لیست mock.  
  - داده در `privacy.store` (Zustand) نگه داشته می‌شود؛ **هیچ API یا اعمال سمت سرور نیست.**
- **Security (در همان صفحه):**  
  - دو ردیف: **Two-Step Verification** و **Passcode Lock** با توضیح "Off" — **فقط برچسب هستند؛ هیچ منطق فعال/غیرفعال یا API ندارند.**
- **Blocked Users:**  
  - لیست کاربران بلاک‌شده در store؛ در چت اگر طرف مقابل بلاک باشد، ارسال پیام غیرفعال و placeholder نشان داده می‌شود.  
  - افزودن به بلاک از داخل چت (منوی پیام/پروفایل) با به‌روزرسانی `blockedUsers` در store؛ **هیچ API بلاک/آنبلاک وجود ندارد.**

**درخواست برای بک‌اند و فرانت:**
- API برای ذخیره/خواندن قوانین حریم خصوصی (شماره، last seen، عکس پروفایل) و لیست بلاک.
- دو قابلیت Two-Step Verification و Passcode Lock را در بک‌اند طراحی کن (و در فرانت اگر لازم است یک صفحه/مودال ساده برای تنظیم فعال/غیرفعال و تأیید اضافه شود).
- فرانت باید لیست بلاک و قوانین حریم را از API بگیرد و در چت و تنظیمات اعمال کند (جایگزین mock و فقط store محلی).

---

### 3.6 دستگاه‌ها و نشست‌ها (Devices / Sessions)

**قابل دسترسی:** در منوی اصلی تنظیمات فعلاً **غیرفعال (حذف شده)** است؛ ولی ویو `DevicesView` هنوز در کد هست.

- **DevicesView:** لیست «Current Session» و «Active Sessions» از `mockedDevices`؛ دکمه «Terminate All Other Sessions» و آیکن خروج هر دستگاه — **هیچ فراخوانی سرویس یا API نیست.**

**درخواست برای بک‌اند و فرانت:**
- مدل session در بک‌اند (مثلاً هر login یک session با device info)؛ API لیست sessions و terminate other sessions.
- وقتی این منو دوباره در تنظیمات فعال شد، فرانت باید از این APIها استفاده کند.

---

### 3.7 ذخیره‌سازی و داده (Storage & Data)

**قابل دسترسی:** در منوی اصلی تنظیمات فعلاً **غیرفعال (حذف شده)** است؛ ویو `StorageView` در کد هست.

- **StorageView:** نمایش عدد ثابت «452 MB» و دسته‌بندی Photos/Videos و Documents/Files؛ ردیف «Clear Entire Cache» فقط `alert("Cache cleared!")` — **هیچ محاسبه یا پاکسازی واقعی نیست.**

**درخواست برای بک‌اند و فرانت:**
- در بک‌اند می‌توان برای رسانه‌های چت حجم ذخیره‌شده را محاسبه کرد و API برای آمار storage ارائه داد.
- در فرانت می‌توان بعداً این API را صدا زد و دکمه Clear Cache را به یک endpoint پاکسازی (در حد امکان) وصل کرد؛ فعلاً می‌توان در spec بک‌اند به عنوان قابلیت بعدی ذکر شود.

---

### 3.8 کمک و درباره (Help)

**قابل دسترسی:** بله — فقط UI.

- ردیف‌ها: Ask a Question، Report a Bug، SpotiChat FAQ، Privacy Policy، Terms of Service — **بدون لینک یا ناوبری.**

**درخواست:** می‌توان در بک‌اند فقط لینک‌های ثابت یا یک endpoint ساده برای متن/لینک این صفحات تعریف کرد؛ فرانت فقط لینک/محتوا را از config یا API بگیرد (اختیاری، با اولویت پایین).

---

### 3.9 چت و مکالمات (Chat / Conversations)

**قابل دسترسی:** بله — UI غنی، داده همه mock.

- **لیست چت:** از `chatService.getChats()` — آرایهٔ mock؛ کلیک برای انتخاب چت و باز کردن چت.
- **استوری:** `chatService.getStories()`؛ ویور استوری با پیشرفت خودکار و ناوبری.
- **ویو چت:**  
  - بارگذاری پیام‌ها با `getMessages(chatId)`.  
  - ارسال متن، پیام صوتی، پیام ویدیویی (long-press با استفاده از `getUserMedia`؛ واقعی در کلاینت، بدون آپلود به سرور).  
  - ریپلای، ری‌اکشن، پین، حذف برای من، حذف برای همه (فعلاً مثل حذف برای من)، فوروارد (مودال placeholder).  
  - جستجو در چت (InChatSearch).  
  - بلاک کاربر از داخل چت (به store حریم خصوصی اضافه می‌شود؛ بدون API).
- **سرویس چت فعلی:** همه در `chat.service.ts` با `delay` و آرایه‌های درون‌حافظه؛ `sendMessage` فقط به mock اضافه می‌کند و بعد از ۲ ثانیه status را به `delivered` تغییر می‌دهد؛ **هیچ WebSocket یا HTTP نیست.**

**درخواست برای بک‌اند:**
- **WebSocket (یا معادل real-time)** برای تحویل و دریافت پیام و به‌روزرسانی وضعیت (sent/delivered/seen).
- APIهای REST برای: لیست چت‌ها، لیست پیام‌ها (با pagination)، ارسال پیام (در صورت نیاز برای fallback یا اولیه).
- **رمزنگاری پیام‌ها و چت‌ها:** طراحی به‌صورت end-to-end یا encryption-at-rest طبق بهترین روش‌های رایج (مثلاً کلید per-chat یا per-user و ذخیرهٔ امن کلیدها).
- مدل داده: User، Chat (یا Conversation)، Message (متن، فایل، صدا، ویدیو، نوع، وضعیت، replyTo، reactions)، و در صورت نیاز Story.

---

### 3.10 تم (Theme)

**قابل دسترسی:** بله — فقط کلاینت.

- Light / Dark / Auto در منوی تنظیمات؛ در `theme.store` و اعمال روی `documentElement` — **نیازی به بک‌اند ندارد** مگر بخواهی ترجیح تم را در پروفایل کاربر ذخیره کنی (اختیاری).

---

### 3.11 خروج (Logout)

**قابل دسترسی:** بله.

- از منوی تنظیمات > Logout؛ دیالوگ تأیید و سپس `auth.store.logout()` و بستن sheet؛ ریدایرکت به `/login`. هیچ فراخوانی API برای invalidation توکن/session نیست.

**درخواست برای بک‌اند:** یک endpoint برای logout (مثلاً باطل کردن refresh token یا session) تا فرانت بعد از logout آن را صدا بزند.

---

### 3.12 گروه و کانال و تماس

- **Create Group / Create Channel:** مودال‌ها در فرانت وجود دارند و از لیست مخاطبین mock استفاده می‌کنند؛ **هیچ API ساخت گروه/کانال یا ارسال پیام گروهی نیست.**
- **تماس صوتی/تصویری:** ویو `CallScreen` و `CallControls` فقط شبیه‌سازی وضعیت و پخش صدا هستند؛ **هیچ WebRTC یا signaling از سرور نیست.**

**درخواست:** در فاز اول می‌توان فقط چت ۱-به-۱ و پیام‌ها را در بک‌اند پوشش داد؛ گروه/کانال و تماس واقعی را می‌توان در فاز بعدی طراحی کرد. در spec بک‌اند ذکر شود که این قابلیت‌ها در roadmap هستند.

## 4. داده و جریان فعلی (فرانت)

- **State:** Zustand برای auth (با persist)، chat (selectedChatId)، modal، theme، settings navigation، privacy.
- **داده:** همه از `auth.service`، `chat.service`، `contact.service` و فایل‌های mock در `settings/mock`.
- **ذخیرهٔ محلی:** فقط auth و یک فلگ permissions و ترجیح صداها.
- **شبکه:** هیچ fetch، axios، WebSocket، یا EventSource استفاده نشده است.

---

## 5. الزامات فنی بک‌اند و دیتابیس

- **Runtime:** Node.js  
- **Framework:** NestJS  
- **Real-time:** WebSocket (مثلاً با Socket.IO یا ws در Nest) برای پیام‌ها و به‌روزرسانی وضعیت (delivered/seen و در صورت نیاز تایپینگ).
- **دیتابیس:** MongoDB برای کاربران، چت‌ها، پیام‌ها، sessions، مخاطبین، تنظیمات حریم خصوصی و اعلان، لیست بلاک، و در صورت نیاز استوری.
- **رمزنگاری:**  
  - پیام‌ها و چت‌ها به‌صورت رمزنگاری‌شده ذخیره/انتقال (تعیین دقیق: E2E با کلید مشترک یا encryption-at-rest با کلید سمت سرور؛ در هر صورت استاندارد و قابل حسابرسی).
- **احراز هویت:**  
  - استاندارد (مثلاً JWT access + refresh یا session-based با httpOnly cookie).  
  - **فعلاً برای تست:** فقط ۳ شمارهٔ از پیش تعریف‌شده مجاز به لاگین باشند و برای همهٔ آن‌ها کد OTP ثابت `12345` پذیرفته شود؛ بعد از این مرحله می‌توان OTP واقعی (SMS یا سرویس دیگر) وصل کرد.

---

## 6. خلاصهٔ کارهایی که باید انجام شود

1. **طراحی و پیاده‌سازی بک‌اند با NestJS + Node.js + MongoDB + WebSocket** برای احراز هویت، پروفایل، مخاطبین، چت، پیام (با رمزنگاری)، sessions، حریم خصوصی و بلاک، اعلان‌ها (ترجیحات).
2. **لاگین:** ۳ شماره دستی + OTP ثابت `12345` در مرحلهٔ فعلی.
3. **تکمیل فرانت در بخش‌های نیمه‌کاره:**  
   - آپلود عکس پروفایل (انتخاب فایل + فراخوانی API).  
   - وصل کردن Contacts در تنظیمات به API.  
   - وصل کردن Privacy و Blocked Users به API.  
   - در صورت فعال‌سازی مجدد منوها: Devices و Storage به API مربوطه.  
   - Two-Step Verification و Passcode Lock: در صورت پیاده‌سازی در بک‌اند، UI سادهٔ فرانت برای فعال/غیرفعال و تأیید.
4. **جایگزینی همهٔ mockها در فرانت** با فراخوانی API و WebSocket واقعی (auth، chat، contact، settings داده‌ای که از سرور می‌آید).

این سند را به‌عنوان پرامپت کامل برای طراحی و ساخت بک‌اند و تکمیل فرانت استفاده کن.
